# Kubernetes DevOps Project with CI/CD

This DevOps project that demonstrates  containerize a static web application using Docker, deploy it on Kubernetes using Minikube, and automate Docker image build and push using GitHub Actions CI/CD.

---

## Project Overview

It includes:

- Docker containerization
- Kubernetes deployment using Minikube
- Namespace-based isolation
- ConfigMap usage
- Service exposure using NodePort
- GitHub Actions CI/CD pipeline
- Docker Hub image push
- Troubleshooting common DevOps issues
- 


## It solves problems like


Eliminates manual application deployment by automating build and deployment using CI/CD.

Solves environment inconsistency issues by containerizing the application using Docker.

Provides scalable application deployment using Kubernetes pods and replicas.

Ensures high availability by running multiple application instances managed by Kubernetes.

Simplifies application updates and version control through Docker image versioning.

Automates image build and registry push using GitHub Actions CI/CD pipeline.

Enables consistent deployment across environments using Kubernetes manifests.

Improves deployment reliability and repeatability using infrastructure-as-code practices.

Demonstrates an end-to-end DevOps workflow from code commit to application deployment.

This project was built to practice core DevOps concepts in a simple and practical way.

### Project Implementation Steps

### 1. Create Project Structure

Create the main project directory and organize folders for the application, Kubernetes manifests, GitHub workflows, and screenshots.

### 2. Create the Application

Create a simple static web application that will be served through a web server inside a container.

### 3. Containerize the Application

Create a Dockerfile and containerize the web application so it can run inside a Docker container.

### 4. Build the Docker Image

Build the Docker image locally using Docker and verify that the image was created successfully.

### 5. Test the Container Locally

Run the container locally and verify that the application works in the browser.

### 6. Start the Kubernetes Cluster

Start a local Kubernetes cluster using Minikube and verify that the cluster is running.

### 7. Create Kubernetes Namespace

Create a namespace to isolate all the resources used for the project from other Kubernetes workloads.

### 8. Create ConfigMap

Create a ConfigMap to store environment configuration that will be used by the application container.

### 9. Create Kubernetes Deployment

Create a Deployment resource that defines the desired number of application pods and manages container lifecycle.

### 10. Deploy Application Pods

Apply the deployment configuration and verify that multiple application pods are created successfully.

### 11. Create Kubernetes Service

Create a Service resource to expose the application running inside the pods.

### 12. Access the Application

Use the service URL generated by Minikube to access the application in a web browser.

### 13. Verify Kubernetes Resources

Verify the health and status of deployments, pods, and services using Kubernetes commands.

### 14. Setup CI/CD Pipeline

Create a GitHub Actions workflow that automatically builds and pushes Docker images when code is pushed to the repository.

### 15. Configure GitHub Secrets

Add Docker Hub credentials as repository secrets to allow the CI/CD pipeline to authenticate and push images.

### 16. Push Code to GitHub

Push the project code to GitHub which triggers the CI/CD pipeline.

### 17. Automatic Docker Image Build

GitHub Actions builds a new Docker image whenever changes are pushed to the repository.

### 18. Push Docker Image to Docker Hub

The CI/CD pipeline pushes the built Docker image to Docker Hub.

### 19. Update Kubernetes Deployment

Update the Kubernetes deployment to use the Docker Hub image.

### 20. Restart Kubernetes Deployment

Restart the deployment so Kubernetes pulls the latest Docker image and creates new pods.

### 21. Verify Application Update

Confirm that the application changes are visible in the browser after the deployment update.
